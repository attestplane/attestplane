# Daily Development Plan for v1.5.8

## Concise Plan

- The v1.5.0→v1.5.8 window built the architecture audit pipeline, plan-to-issues workflow, and auto-loop scheduling. All 14 real commits are infrastructure: planning scripts, CI dispatch, issue sync, and git proxy hardening.
- The deterministic fallback plan (generated by `architecture_audit_trigger.py`) proposes generic product work, but this window has zero product‑facing SDK, verifier, or conformance changes. Opus should instead produce a plan that:
  1. Closes the feedback loop by feeding the new planning pipeline a real conformance fixture increment — this validates the pipeline end‑to‑end with minimal risk.
  2. Pins a `verify --json` output‑contract fixture at the conformance level so the plan‑to‑issues workflow can consume it.
  3. Documents the new daily‑plan workflow in runbooks.
- No P0 issues: this is a daily patch with no breaking changes or product milestones.

## P1 Issues

### ISSUE 1 · [P1][verifier][conformance] Pin `verify --json` output‑contract fixture for CI

Owner: verifier/conformance

Affected modules:

- Python CLI verify JSON serialization
- Conformance fixture set
- Fixture‑lock maintenance

Acceptance criteria:

1. Extract the deterministic `verify --json` output for a minimal valid proof bundle (single event, no anchoring) as a versioned conformance fixture under `tests/conformance/`.
2. The fixture records the full JSON payload including `schema_version`, `result`, `exit_code`, `reason_code`, `reasons`, `bundle.schema_version`, `anchoring.status`, and `taxonomy_version`.
3. A conformance test asserts that `verify --json` against the same bundle produces byte‑identical output to the pinned fixture.
4. Update `FIXTURE_HASHES.lock` with the new fixture hash.
5. Reference the existing verify‑JSON test module rather than duplicating its scope.

Validation commands:

- `cd sdk/python && python3.11 -m pytest tests/conformance -k 'verify_json or output_contract' -q`
- `cd sdk/python && python3.11 scripts/conformance/verify_fixture_lock.py`
- `cd sdk/typescript && npm test -- --runInBand`
- `git diff --check`

Rollout / migration notes:

- The fixture must be forward‑compatible: additive‑optional `schema_version` fields should not invalidate it.
- Do not regenerate unrelated fixture hashes.

### ISSUE 2 · [P1][test][conformance] Validate the daily‑plan cross‑SDK output contract

Owner: conformance

Affected modules:

- Python conformance vectors
- TypeScript CLI tests
- Cross‑SDK roundtrip suite

Acceptance criteria:

1. Add a cross‑SDK vector that runs `verify --json` on both Python and TypeScript SDKs against the same proof bundle and confirms the JSON outputs are structurally equivalent (same keys, same value types, same exit codes).
2. The vector is wired into the existing cross‑SDK roundtrip test (`tests/cross_sdk/`).
3. Both SDKs produce the same `taxonomy_version`, `reason_code` values for the same input.
4. Reference the existing conformance issues rather than duplicating scope.

Validation commands:

- `cd sdk/python && python3.11 -m pytest tests/conformance -k 'cross_sdk or roundtrip' -q`
- `cd sdk/python && python3.11 -m pytest tests/cross_sdk/ -q`
- `cd sdk/typescript && npm test -- --runInBand`
- `git diff --check`

Rollout / migration notes:

- This verifies that the plan‑to‑issues pipeline can consume real conformance work.
- Keep the vector minimal — one bundle, two SDKs, same expected JSON shape.
