# v1.7.x User-Visible Delta

This support-task delta extends Issue #157. It records the verifier, CLI,
and schema-version surface that became visible in the v1.7.x train without
replacing the product work tracked in the source issues.

## Changes Since #157

The verifier UX for ISSUE 1/2 is split into human-readable and
machine-readable paths:

| Surface | Behavior | Canonical reference |
|---|---|---|
| `verify --explain` | Human-oriented failure summary for operators. It is additive to `verify --json`: when both flags are present, the rejection text is carried in `reasons[].explanation` without changing `schema_version`, while `reasons[].message` keeps the failure detail. | [`docs/cli/verify-json.md`](../cli/verify-json.md) |
| `verify --json` | Machine-readable verifier result. The JSON payload includes `schema_version`, `result`, `exit_code`, `reasons[]`, and `bundle.digest`. This is the #220 surface. | [`docs/cli/verify-json.md`](../cli/verify-json.md) |
| JSON schema location | The JSON report contract is documented separately from the bundle wire-format schema. The #217 schema-version policy lives in [`docs/schema/verify-json.md`](../schema/verify-json.md). | [`docs/schema/verify-json.md`](../schema/verify-json.md) |
| Reason-code table | The stable `att.verify.*` rejection taxonomy lives in [`docs/errors.md`](../errors.md). | [`docs/errors.md`](../errors.md) |

- Reason-code taxonomy reference: `docs/errors.md` now documents the SDK-public
  `att.verify.*` rejection reason surface introduced by Issue #172.
- `verify --explain` and `verify --json` preserve the same exit-code contract:
  use `--explain` when a human needs the summary, and use `--json` when a CI
  job needs to branch on machine-readable output.
- `verify --explain` is additive only. It does not bump `schema_version`, and
  it does not create a new verifier schema or bundle contract.
- `schema_version` policy: bundle schema versions remain separate from the
  verifier reason-code taxonomy. Missing versions surface as
  `att.verify.schema_version_missing`, and unsupported versions surface as
  `att.verify.schema_version_unsupported`.
- Major-version-ahead bundles are now explicitly rejected, and fail-closed
  critical/required fields surface `att.verify.schema_unknown` instead of
  slipping through additive acceptance.
- Forward-compatible additive fields: bundles that only add unknown fields are
  accepted by the verifier. This is the #217 additive rule: additive unknown
  fields do not flip `ok` when the rest of the bundle is valid.
- Negative cases remain strict: required-field, structure, and signature
  failures still map to the relevant `att.verify.*` rejection reasons in the
  canonical taxonomy.
- Migration note: CI consumers should branch on the process exit code first,
  then on `.result` and ordered `reasons[]` entries.
- negative conformance vectors: see `fixtures/negative/` and
  `tests/conformance/test_verify_json_schema.py` for the failure cases that
  pin canonicalization, schema, and structured reason behavior.

### Reason-Code Table

The release note only names the codes that matter for this delta. The full
taxonomy remains in `docs/errors.md`.

| Code | Meaning |
|---|---|
| `att.verify.schema_version_missing` | A known bundle, payload, signature, or verifier schema version is missing. |
| `att.verify.schema_version_unsupported` | A known bundle, payload, signature, or verifier schema version is recognized but unsupported. |
| `att.verify.required_field_missing` | A required top-level, nested, signature, or verifier-envelope field is absent. |
| `att.verify.signature_missing` | Strict verification requires signature material but none is present. |
| `att.verify.structure_invalid` | Known bundle relationships are malformed, duplicated, dangling, or out of order. |
| `att.verify.schema_invalid` | The input shape is malformed for a known verifier schema. |

## CI Gating Example

Use the exit code as the gate and inspect `.result` plus `reasons[]` from the
JSON payload. This example keeps the shell branch on the verifier exit status,
then uses `jq` to read the machine-readable reason surface:

```sh
attestplane verify --json "$bundle" > verify.json
rc=$?
result=$(jq -r '.result' verify.json)

if [ "$rc" -eq 0 ] && [ "$result" = "pass" ]; then
  exit 0
fi

jq -r '.reasons[]? | "\(.code) \(.path) \(.message)"' verify.json
exit "$rc"
```

In this pattern, `reasons[]` is the derived list of verifier rejection
reasons and `.bundle.digest` pins the verified input bytes.

### `verify --explain` Addendum

The append-only `verify --explain` surface stays aligned with the #220 JSON
output and the #217 schema-version policy. It is for operators who need a
plain-language rejection summary alongside the structured CI contract. When
combined with `--json`, the stable rationale string is exposed in
`reasons[].explanation`.

Synopsis:

```sh
attestplane verify --json --explain "$bundle"
```

Pass example:

```json
{
  "schema_version": 1,
  "result": "pass",
  "exit_code": 0,
  "reasons": [],
  "bundle": {
    "schema_version": 1,
    "digest": "..."
  }
}
```

Fail example:

```json
{
  "schema_version": 1,
  "result": "fail",
  "exit_code": 1,
  "reasons": [
    {
      "code": "att.verify.schema_version_unsupported",
      "path": "bundle.schema_version",
      "message": "bundle schema_version 2 is not supported"
    }
  ],
  "bundle": {
    "schema_version": 2,
    "digest": "..."
  }
}
```

The failure example uses the shared `att.verify.*` taxonomy documented in
[`docs/errors.md`](../errors.md), and the `verify --explain` surface itself
does not bump `schema_version`.
