# v0.0.2-alpha — release notes draft

> **Status:** DRAFT. Prepared from main commit history; awaiting
> maintainer approval before tagging. When tagged, this file is moved
> to the published GitHub Release body and replaced by a brief
> "v0.0.2-alpha shipped" pointer.

## Headline

Second public alpha. Adds the v0.1 design surface (boundary ADR + event
taxonomy + attestation gates), verifier predicates + CLI, a
JSONL storage backend, obligation registries for EU AI Act Article 12
and DORA Article 8, and a second cross-language canonicalization
primitive (canonical-text). v0.0.1-alpha's hash-chain contract is
preserved bit-for-bit — every existing chain continues to verify.

This is an alpha substrate release: not production-ready, not compliance-ready,
and not a legal/regulatory certification. The current
`attestplane verify` command is chain/report-oriented: it replays
bundle events, compares the embedded `verification_report` with the
recomputed chain result, and fails closed on malformed ProofBundle
metadata and `policy_trace_refs` closure. It does not perform full
ProofBundle, signature, anchor, or compliance certification
verification.

## What you can do today that you couldn't with v0.0.1-alpha

```bash
pip install attestplane==0.0.2a0
attestplane doctor              # environment self-check
attestplane inspect chain.jsonl # chain summary + verify
attestplane export chain.jsonl --out bundle.json
attestplane verify bundle.json  # chain/report-oriented check only
```

Chain -> export -> bundle -> chain/report-oriented verify is now a real
local workflow. It must not be described as full ProofBundle, signed,
anchored, or compliance verification.

```typescript
import { ProofBundleBuilder, verifyProofBundle } from '@attestplane/attestplane';
// TypeScript SDK reaches API parity with Python for proof-bundle work;
// cross-language byte identity proven by smoke test in the release CI.
```

## Three concrete claim-status changes

These should propagate into [`docs/policy/allowed_claims.md`](../policy/allowed_claims.md)
upon release:

1. **EU AI Act Article 12 obligation registry** moves from
   "shipped at M5" to **`field_supported` for four entries**
   (`art12.3a` period of use, `art12.3b` reference database,
   `art12.3c` matched input, `art12.3d` human verifier).
2. **DORA Article 8 obligation registry** moves from "designed
   toward, ships at M5" to **`designed_toward` for four entries +
   `field_supported` for `art8.5`** (privileged-access recording).
3. **Verifier predicates and CLI** move from "planned for M5" to
   **shipped with restricted scope**: `verify_proof_bundle()` library
   + `attestplane verify <bundle.json>` work today for chain/report
   agreement, ProofBundle metadata closure, and `policy_trace_refs`
   closure. They do not perform full ProofBundle, signature, anchor,
   or compliance certification verification.
4. **Py/TS conformance and public import surfaces** were hardened for
   the final alpha sweep: TypeScript canonical edge cases now fail
   closed, payload validators reject schema drift consistently, shared
   negative fixtures replay in both SDKs, and package import smoke
   tests cover the documented root exports.

## What stays the same as v0.0.1-alpha

- The substrate's `event_hash` computation, `prev_hash` linkage,
  canonical-JSON byte determinism — unchanged. The frozen
  [`vectors.json`](https://github.com/attestplane/attestplane/blob/main/sdk/python/tests/conformance/vectors.json) ten-vector
  contract continues to verify against the v0.0.2-alpha code on every
  CI run.
- RFC-3161 anchoring and signing sidecar primitives exist in the
  source tree, but the default CLI verifier does not perform anchor or
  signature checks.
- The substrate remains in-memory by default; the new
  `JsonlStorageBackend` is a separate opt-in.

## What's NOT in v0.0.2-alpha

- RFC-3161 TSA anchoring implementation (still M5 per ADR-0003).
- Sigstore / Rekor redundant anchor (M6 per ADR-0006).
- Multi-writer / SQLite / Postgres storage backend (still anticipated
  storage-backend ADR).
- Rust SDK (M7).
- Production PyPI publication. v0.0.2-alpha publishes to TestPyPI
  (sandbox) only, same as v0.0.1-alpha.

## Supply chain

The release follows the same hygiene path as v0.0.1-alpha:

- npm: `npm publish --provenance` → Sigstore transparency log entry.
- TestPyPI: OIDC trusted publishing (no long-lived tokens).
- GitHub Release assets (wheel + sdist + npm tarball + CycloneDX SBOM
  JSON + XML) signed with Sigstore keyless cosign.
- All third-party GitHub Actions SHA-pinned. CodeQL + OSV-Scanner +
  OSSF Scorecard + reproducible-wheel-build gates remain on every
  push.

## Verification

A consumer pulling `0.0.2a0` from TestPyPI and `@0.0.2-alpha.0` from
npm SHOULD verify, in order:

1. **Sigstore provenance** of the npm tarball against the
   [Sigstore transparency log](https://search.sigstore.dev/).
2. **Cosign signature** of the GitHub Release assets against the
   public key embedded in the cosign signature bundle.
3. **REUSE compliance** of the source tree against
   [REUSE 3.3](https://reuse.software/).
4. **End-to-end chain integrity** by running the substrate's own
   conformance vectors through both SDKs:

   ```bash
   pip install attestplane==0.0.2a0 && python -m pytest \
     --pyargs attestplane.tests.test_conformance
   ```

   (or the equivalent vitest invocation against the npm tarball).

## Boundaries

- The OSS substrate does NOT contain an AIOS adapter implementation.
  `aios_spec.py` is docstring-only by design. The concrete AIOS
  adapter lives in the AIOS commercial repository per
  [ADR-0004 § 4](../adr/0004-aios-to-attestplane-boundary.md).
- The OSS substrate does NOT own any execution-plane authority —
  no lease issuance, no budget enforcement, no settlement execution,
  no scheduling. These remain in the runtime that produces events.
  See [ADR-0004 § 2](../adr/0004-aios-to-attestplane-boundary.md) for
  the ten enumerated boundary cases.
