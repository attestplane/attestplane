# v1.10.1

`v1.10.1` is a **Controlled Availability (CA)** patch cut on the `v1.x` pre-GA
line. CA is an availability channel expressed through this release note and the
npm dist-tag, not a SemVer suffix; the package version is the suffix-free
`1.10.1` (see [`ga-ca-cut-criteria.md`](../release/ga-ca-cut-criteria.md) and
[`npm-dist-tag-policy.md`](../release/npm-dist-tag-policy.md)).

CA is **not** GA, **not** a production-readiness claim, **not** a regulatory
certification, and **not** a legal-compliance claim (see Explicit Boundaries).

## Changes Since Previous Stable (1.10.0)

- **Version self-report fixed (release integrity).** The published `1.10.0`
  packages self-reported `1.8.4` — `attestplane.__version__` / the CLI
  `--version` (PyPI) and `VERSION` (npm) were hand-maintained literals the
  release train never bumped, so they drifted three minor versions behind the
  package metadata. On the npm side this stale value flowed into proof-bundle
  `producer_runtime` provenance.
- Both SDKs now **derive** their runtime version from package metadata
  (pyproject `version` / `package.json`) as a single source of truth, so the
  self-reported version can never again drift from the published artifact.
- The import-surface tests assert the consistency invariant (not a literal),
  and `release-cd` now rejects reintroduction of a hardcoded version literal in
  either SDK's version source.

## npm Dist-Tag Decision

- `latest` → `1.10.1`
- `ca` → `1.10.1` (Controlled Availability channel marker)
- `beta` and `rc` are unchanged by this cut.

Moving `latest` to `1.10.1` is the intended consumer-facing default for this CA
line; it is a stable package version, not a prerelease. RC packages continue to
publish under `rc` and must not move `latest`.

## Installation

```sh
# PyPI (Python)
pip install attestplane==1.10.1

# npm (TypeScript) — pin the version explicitly
npm install @attestplane/attestplane@1.10.1
```

## Verification (clean-environment smoke)

Clean-install smoke and cross-SDK roundtrip for the published packages run via
the read-only `ca-postpublish-verify` workflow (dispatch with
`version=1.10.1`). The same checks can be reproduced locally:

```sh
# Python: install the published wheel into a clean venv, then smoke it.
python -m venv /tmp/ca && /tmp/ca/bin/pip install "attestplane==1.10.1"
/tmp/ca/bin/python scripts/release/ca_registry_smoke.py smoke 1.10.1

# TypeScript: install the published package into a throwaway project, then smoke it.
mkdir /tmp/ca-ts && cd /tmp/ca-ts && npm init -y && npm install "@attestplane/attestplane@1.10.1"
cp <repo>/scripts/release/ca_registry_smoke.mjs . && node ca_registry_smoke.mjs smoke 1.10.1
```

Each smoke asserts the install resolves to the published package, that its
self-reported version equals `1.10.1` (the drift guard), that a freshly built
bundle verifies, and that a tampered bundle is rejected.

## Rollback

Rollback owner: **[MAINTAINER DECISION REQUIRED]**.

Registry correction follows
[`npm-dist-tag-policy.md` → Manual Recovery](../release/npm-dist-tag-policy.md):
do not delete or reuse a published version; record the observed registry state;
move the dist-tag back through the dedicated `manage-npm` path; and add a
release-note or validation entry explaining the correction. PyPI cannot
overwrite a released version; a bad cut is superseded by the next patch.
