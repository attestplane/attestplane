# ADR-0005 T6 TypeScript Mirror — Architect Review (2026-05-17)

> **Status:** read-only design walkthrough by an `opus-architect`
> terminal-review-gate dispatch after T1–T5 + T7 shipped (commit
> `0fc0776`). Locks 10 implementation-critical decisions for the
> TypeScript mirror of the event-signing scheme.
> Supplements [`adr_0005_signing_plan_20260517.md`](./adr_0005_signing_plan_20260517.md)
> and [`adr_0005_t3_t4_review_20260517.md`](./adr_0005_t3_t4_review_20260517.md).

## 1. Locked decisions

### Decision 1 — TS module layout: `sdk/typescript/src/signing/` subdirectory

5 files mirroring Python 1:1 (`base.ts` / `providers.ts` / `signer.ts`
/ `trust_roots.ts` / `verifier_ext.ts`). Flat re-export through
`index.ts` preserves external API surface.

### Decision 2 — Ed25519 via `node:crypto`: JWK seed path + SPKI export

Node 22 baseline. `crypto.createPrivateKey({format:'jwk', key:{kty:'OKP', crv:'Ed25519', d:base64url(seed)}})`
turns a 32-byte seed into a `KeyObject`. SPKI export via
`createPublicKey(privateKey).export({format:'der', type:'spki'})` (the
intermediate `createPublicKey` is the stable API). `crypto.sign(null, ...)` /
`crypto.verify(null, ...)` for Ed25519 (algorithm=null per Node docs).

### Decision 3 — TrustRoots: JSON only

PyYAML is Python-side convenience for operators; TS SDK consumes JSON
trust-roots files. Same schema fields exactly; operators convert YAML
to JSON via `yq` if needed. **Rationale**: avoids adding a 50 KB yaml
dep to the TS package whose only runtime dep today is `uuid`. Future
opt-in `loadTrustRootsYaml()` is a follow-up if a customer requires it.

### Decision 4 — KeyProvider subset

Ship in v1: `InMemoryKeyProvider`, `MultiSignerProvider`,
`FileKeyProvider`, `EnvKeyProvider`. The latter two are Node-only;
JSDoc warns they require `node:fs` / `process.env`. No physical split
(browser build is not a v0.0.2-alpha target).

### Decision 5 — Signer worker: sync-only

`signEvent()` + `signSegmentHead()` synchronous APIs cover 100% of
conformance vectors. Background mode (`Anchorer`-style) deferred to a
future ticket — TS `anchoring.ts` already lacks the worker pattern; no
need to break parity here.

### Decision 6 — Naming: snake_case JSON fields + camelCase functions

Matches existing TS pattern (`AnchorRecord.tsa_provider_id` etc.).
`SignatureRecord` interface fields = snake_case; functions = camelCase
(`verifyChainWithSignatures`, `loadTrustRoots`, `deriveKeyId`).

### Decision 7 — Plurality + coverage transitivity: exact mirror

`STATUS_RANK = Object.freeze({ valid: 0, expired_key: 1, invalid: 2,
unknown_key: 3, unsigned: 4 })`. `mergeStatusAtSeq` picks rank-minimum
per seq. `computeSignedSegmentCount` applies option-a transitivity:
valid segment-head at seq=N covers seqs {prev_signed_head+1..N};
per_event covers `signed_seq` only.

### Decision 8 — Conformance gate: 16 vitest assertions

See § 4. Asserts include byte-equal canonical payload reconstruction,
SPKI byte-equal across Python/TS, key_id derivation, end-to-end sign
round-trip equality.

### Decision 9 — Out-of-scope (explicitly)

- Background-thread Signer (decision 5).
- TS Sigstore Rekor port.
- ADR-0007 Fulcio/OIDC hooks.

In-scope additions (clarified from earlier):

- `proof_bundle.ts` extension to carry `signatures?` field.
- `verifyChainFull` TS port (otherwise `BundleVerificationResult` shape is asymmetric).

### Decision 10 — R1 risk concretised + test

Ed25519 SPKI per RFC 8410 is fixed-length 44 bytes with constant
prefix (`30 2a 30 05 06 03 2b 65 70 03 21 00 <32-byte-key>`). No
leading-zero ambiguity. Conformance assertion #3 + #15 explicitly
catch any byte mismatch in pubkey DER or derived key_id.

## 3. `node:crypto` recipes (load-bearing)

### Seed → KeyObject

**Implementation revision (2026-05-17 PM)**: the initial JWK recipe
below fails on Node 22 because `createPrivateKey({format:'jwk', key:
{kty:'OKP', crv:'Ed25519', d}})` requires the public `x` field
alongside `d`. Recomputing `x` from `d` requires a third-party Ed25519
implementation. The PKCS#8 wrap path below sidesteps the issue and is
byte-equal to Python `cryptography`'s
`Ed25519PrivateKey.from_private_bytes(seed)` serialisation.

Implemented recipe (RFC 8410 PKCS#8 wrap):

```ts
import { createPrivateKey, KeyObject } from 'node:crypto';

const PKCS8_ED25519_PREFIX = Buffer.from(
  '302e020100300506032b657004220420', 'hex',
);

function seedToPrivateKey(seed: Uint8Array): KeyObject {
  if (seed.length !== 32) throw new KeyProviderError('seed must be 32 bytes');
  const pkcs8 = Buffer.concat([PKCS8_ED25519_PREFIX, Buffer.from(seed)]);
  return createPrivateKey({ key: pkcs8, format: 'der', type: 'pkcs8' });
}
```

ASN.1 decode of the 16-byte prefix:

```
SEQUENCE (46 bytes)            -- 30 2e
  INTEGER 0                     -- 02 01 00
  SEQUENCE (5 bytes)            -- 30 05
    OID 1.3.101.112             -- 06 03 2b 65 70   (id-Ed25519)
  OCTET STRING (34 bytes)       -- 04 22
    OCTET STRING (32 bytes)     -- 04 20
      <seed bytes>              -- (caller supplies)
```

Original (rejected) JWK recipe — kept for historical context only:

```ts
function seedToPrivateKey(seed: Uint8Array): KeyObject {
  // ❌ fails: Node 22 requires `x` alongside `d`.
  const d = Buffer.from(seed).toString('base64url');
  return createPrivateKey({
    key: { kty: 'OKP', crv: 'Ed25519', d },
    format: 'jwk',
  });
}
```

### KeyObject → SPKI DER

```ts
import { createPublicKey } from 'node:crypto';

function exportPublicKeyDer(privateKey: KeyObject): Uint8Array {
  const pub = createPublicKey(privateKey);
  return new Uint8Array(pub.export({ format: 'der', type: 'spki' }));
}
```

### Sign / verify

```ts
import { sign, verify } from 'node:crypto';

ed25519Sign:  sign(null, payload, privateKey)
ed25519Verify: verify(null, payload, publicKey, signature)  // returns bool
```

### `deriveKeyId`

```ts
import { createHash } from 'node:crypto';

export function deriveKeyId(publicKeyDer: Uint8Array): string {
  if (publicKeyDer.length === 0) throw new SigningError('public_key_der empty');
  return createHash('sha256').update(publicKeyDer).digest().subarray(0, 16).toString('hex');
}
```

### `buildSegmentHeadPayload`

```ts
import { canonicalize } from '../canonical.js';
import { SCHEMA_VERSION } from '../hashchain.js';

export function buildSegmentHeadPayload(chainId: string, head: ChainHead): Uint8Array {
  if (!chainId) throw new SigningError('chain_id must be non-empty');
  return canonicalize({
    chain_id: chainId,
    event_hash: Buffer.from(head.event_hash).toString('hex'),
    schema_version: SCHEMA_VERSION,
    seq: head.seq,
    signature_schema_version: SIGNATURE_SCHEMA_VERSION,
  });
}
```

## 5. Risk register (T6-specific top 3)

| # | Risk | Mitigation |
|---|------|-----------|
| R1 | Node `KeyObject.export({format:'der', type:'spki'})` API behaviour varies | Use `createPublicKey(privateKey).export(...)` intermediate (stable). Require Node ≥ 22. Assertion #3 + #15 fail fast on drift. |
| R2 | JSON `import ... assert/with { type: 'json' }` unstable across Node 22 minors | Use `readFileSync(...) + JSON.parse(...)`. No build config dependency. |
| R3 | `canonicalize` per_event AuditEvent input shape differs Py vs TS (timestamp Date vs string) | `buildPerEventPayload` JSDoc requires `timestamp` already RFC-3339-µs-Z string. Matches existing convention in `_serialize_event`. |
